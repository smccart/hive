# Configuration

HiveScan is zero-config by default. Point it at a directory and it auto-discovers all projects — monorepo workspaces, standalone repos, anything with a `package.json` or `.git`.

## CLI options

```bash
hivescan [options]
```

| Option | Default | Description |
|--------|---------|-------------|
| `-d, --dir <path>` | Parent of current repo | Directory to scan for projects (repeatable) |
| `-p, --port <number>` | `4269` | Server port for the web UI |
| `--poll <seconds>` | `5` | Port scan interval |
| `--no-open` | — | Don't open the browser automatically |

### Examples

```bash
# Scan the current repo and its siblings
hivescan

# Scan a specific directory
hivescan --dir ~/Sites

# Multiple directories
hivescan --dir ~/work/frontend --dir ~/work/backend

# Custom port and slower polling
hivescan --port 5000 --poll 15
```

## Auto-discovery

HiveScan discovers projects in two ways:

### Directory scanning

On startup, HiveScan scans the target directory (and its immediate children) for projects. A directory is considered a project if it contains a `package.json` or `.git` directory.

For monorepos, HiveScan also scans these subdirectories for workspace children:

- `apps/`
- `packages/`
- `services/`
- `libs/`
- `modules/`

Config/tooling packages (eslint, prettier, tsconfig, etc.) are automatically excluded.

### Port scanning

Every few seconds (configurable with `--poll`), HiveScan scans for listening TCP ports on your system using `lsof`. It resolves each port's process to a project directory and annotates matching projects with their live ports. This is how port links appear in the UI header.

System services (databases, Docker, SSH, etc.) are automatically filtered out.

## Data storage

Session data is stored in `~/.hive/`:

```
~/.hive/
  projects/
    <project-id>/
      agent.buf     # terminal history (rolling, max 50 KB)
```

Project IDs are derived from the absolute path (e.g., `/Users/you/Sites/myapp/apps/web` becomes `users-you-sites-myapp-apps-web`).

## Model selection

HiveScan defaults to `claude-sonnet-4-6`. You can switch models from the sidebar dropdown — this affects newly spawned agents only. Available models:

- Opus 4.6 (`claude-opus-4-6`)
- Sonnet 4.6 (`claude-sonnet-4-6`)
- Haiku 4.5 (`claude-haiku-4-5-20251001`)

## Claude Code permissions

HiveScan can manage each project's `.claude/settings.json` file, which controls how Claude Code handles tool permissions. Edit from the dashboard UI (click **Permissions** in the sidebar) or by hand.

```json
{
  "defaultMode": "acceptEdits",
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git diff *)",
      "Bash(git commit *)",
      "Read"
    ],
    "deny": [
      "Bash(git push *)",
      "Bash(rm -rf *)",
      "Read(.env*)"
    ]
  }
}
```

### Default modes

| Mode | Behavior |
|------|----------|
| `default` | Prompts for each tool on first use |
| `acceptEdits` | Auto-approves file edits, prompts for shell commands |
| `plan` | Read-only — analyze but don't modify |
| `dontAsk` | Auto-denies unless listed in allow rules |
| `bypassPermissions` | Skips all prompts (isolated environments only) |

### Rule syntax

Rules follow the pattern `Tool(pattern)` where `Tool` is `Bash`, `Edit`, `Read`, or `Write`, and `pattern` is a glob:

- `Bash(npm run *)` — matches any `npm run` command
- `Edit(/src/**)` — matches edits to any file under `src/`
- `Read` — matches all file reads (no pattern = match all)
- `Read(.env*)` — matches reads of `.env`, `.env.local`, etc.

Deny rules take priority over allow rules.

## Environment

HiveScan resolves Claude's path and your login-shell `PATH` automatically, so tools like `nvm`, `fnm`, and `mise` work as expected inside the spawned agent processes.

Only safe environment variables are forwarded to spawned Claude processes (HOME, PATH, LANG, NODE_ENV, ANTHROPIC_API_KEY, etc.). Sensitive variables from the parent process are not leaked.

## Platform support

HiveScan supports macOS and Linux. Windows is not currently supported due to dependency on Unix-specific tools (`lsof`, `/bin/zsh`, `/bin/bash`).
